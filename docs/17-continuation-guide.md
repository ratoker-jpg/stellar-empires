# AI Continuation Guide

**Статус:** Accepted  
**Последнее обновление:** 2026-07-19

## 1. Назначение

Этот документ нужен, чтобы любой новый AI-чат мог продолжить Stellar Empires без доступа к приватной истории предыдущего разговора.

Репозиторий:

```text
ratoker-jpg/stellar-empires
```

Основная ветка:

```text
main
```

Публичная сборка:

```text
https://ratoker-jpg.github.io/stellar-empires/
```

## 2. Обязательный порядок начала новой сессии

До планирования или изменения кода:

1. прочитать корневой `AGENTS.md`;
2. прочитать этот документ;
3. прочитать `docs/project-status.json`;
4. прочитать `docs/16-execution-roadmap.md`;
5. прочитать `docs/roadmap-pr-index.json`;
6. проверить фактический `main`;
7. проверить последние смерженные PR после `lastMergedPr` из status-файла;
8. при расхождении сначала исправить status/handoff, опираясь на GitHub как источник истины.

Нельзя начинать разработку только по памяти модели или по старому сообщению пользователя.

## 3. Текущее подтверждённое состояние

На момент создания этого документа:

- работают ресурсы, энергия, население, стабильность, склады и очередь строительства;
- существует P1-набор production-ассетов «Эгиды»;
- канонический roadmap расписан до Release 1.0.

Фактические детали всегда сверяются с `docs/project-status.json` и последними PR.

## 4. Правило размера delivery batch

Стандартный пакет — **6 последовательных PR**.

AI-исполнитель может без дополнительного согласования расширить пакет до **7–8 PR**, когда дополнительные задачи:

- являются прямым продолжением того же этапа;
- уже предусмотрены roadmap;
- имеют понятные зависимости;
- остаются независимо проверяемыми;
- не содержат нерешённых рисков архитектуры, данных, лицензий или инфраструктуры;
- выполняются после стабильного шестого PR.

Нельзя завершать обычный пакет на четырёх или пяти PR только потому, что старое минимальное правило было выполнено.

## 6. Стандартный GitHub workflow

Для каждого PR:

1. создать новую ветку от актуального `main`;
2. реализовать одну проверяемую цель;
3. добавить или обновить тесты;
4. обновить документацию и schema migration при необходимости;
5. открыть PR;
6. дождаться CI;
7. самостоятельно исправить lint/typecheck/test/build failures;
8. проверить diff и отсутствие известных блокеров;
9. squash-merge;
10. следующую ветку создавать только от нового `main`.

Стандартный CI:

```text
lint → typecheck → tests → production build
```

## 7. Инварианты проекта

- Симуляция не зависит от DOM и Phaser.
- `Math.random()` не используется в simulation core; случайность получает seed.
- Игровое время не читается напрямую из системных часов внутри домена.
- Игрок и боты используют одинаковые команды и ограничения.
- Боты не читают скрытое состояние игрока.
- Событие не выполняется дважды.
- Один корабль не может одновременно находиться в двух местах.
- Ресурсы не становятся отрицательными.
- Любое несовместимое изменение save state получает новую версию, migration и fixture-тест.
- UI не является источником игрового состояния.
- Production-ассеты должны быть оригинальными или иметь ясные права.

## 8. Полномочия и границы

Разрешено без дополнительного подтверждения:

- создавать ветки, коммиты и PR;
- исправлять CI;
- переводить draft PR в ready;
- squash-мержить зелёные PR;
- продолжать roadmap стандартным пакетом из шести PR;
- расширять пакет до восьми по правилам выше;
- обновлять status и continuation docs.

Запрещено без отдельного подтверждения:

- удалять репозиторий или `main`;
- force-push в общие/защищённые ветки;
- отключать проверки ради merge;
- мержить известный сломанный или destructive код;
- публиковать секреты и персональные данные;
- добавлять ассеты с неясными правами;
- менять видимость репозитория;
- подключать платные сервисы или инфраструктуру.

## 9. Что обновлять после каждого PR и batch

После PR обновить `docs/project-status.json`, когда изменились:

- `lastMergedPr`;
- `lastMergeSha`;
- `nextPr`;
- текущий milestone;
- состав следующего batch;
- ключевая работающая механика;
- известные ограничения.

После batch проверить и при необходимости обновить этот документ:

- фактическое состояние проекта;
- следующий стандартный пакет из шести PR;
- допустимые седьмой и восьмой PR;
- риски и внешние блокеры.

## 10. Формат итогового сообщения владельцу

После выполнения batch сообщить:

- какие PR смержены;
- краткий результат каждого;
- что теперь доступно в игре;
- какие проверки прошли;
- известные ограничения;
- номера следующего стандартного пакета;
- было ли использовано расширение до 7–8 PR.

Не утверждать, что работа продолжится в фоне после ответа.
