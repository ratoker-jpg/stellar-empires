# Development Rules

**Статус:** Draft  
**Последнее обновление:** 2026-07-18

## 1. Цель

Правила нужны, чтобы проект не превращался в набор связанных патчей без проверяемой архитектуры.

Приоритеты:

1. корректность состояния;
2. воспроизводимость;
3. простота изменения;
4. производительность;
5. визуальная полировка.

## 2. Работа через Git

Основная ветка: `main`.

Изменения выполняются в ветках:

```text
agent/<краткое-название>
feature/<краткое-название>
fix/<краткое-название>
docs/<краткое-название>
```

Каждая существенная задача оформляется отдельным Pull Request.

Прямые изменения в `main` допустимы только для первоначальной инициализации или аварийного исправления инфраструктуры.

## 3. Pull Request

PR должен содержать:

- что изменено;
- зачем;
- какие системы затронуты;
- как проверено;
- риски;
- что намеренно не сделано.

Большой PR нужно разбивать, если изменения можно независимо проверить и влить.

## 4. Коммиты

Коммит должен описывать одно логическое изменение.

Хорошо:

```text
Add deterministic event queue
Fix fleet return duplication
Document save migration rules
```

Плохо:

```text
updates
fix
final changes
```

## 5. Definition of Done

Механика считается готовой, когда:

- правила описаны;
- структура данных определена;
- команда или событие реализованы;
- ошибки возвращаются в структурированном виде;
- состояние сохраняется;
- UI отображает результат;
- бот умеет использовать механику либо явно исключён до своего этапа;
- добавлены тесты;
- документация обновлена;
- production-сборка проходит.

## 6. TypeScript

Требования:

- строгий режим;
- без неявного `any`;
- `unknown` вместо небезопасного `any` на границах;
- явные типы публичных API;
- discriminated unions для команд и событий;
- неизменяемые входные DTO там, где это практично;
- проверка данных при импорте и загрузке конфигурации.

Не использовать типы как замену runtime-валидации внешних данных.

## 7. Доменная логика

Доменная логика должна быть чистой настолько, насколько это возможно.

Функция расчёта не должна:

- обращаться к DOM;
- читать текущее системное время напрямую;
- вызывать `Math.random()`;
- читать IndexedDB;
- создавать Phaser-объекты;
- отправлять сетевые запросы;
- изменять глобальные переменные.

Все зависимости передаются явно.

## 8. Ошибки

Ожидаемые игровые отказы не являются исключениями.

Пример:

```ts
type CommandResult<T> =
  | { ok: true; value: T }
  | { ok: false; code: string; message: string; details?: unknown };
```

Исключения используются для действительно неожиданных ошибок и нарушений инвариантов.

## 9. Тесты

Минимальные уровни:

### Unit

- формулы;
- проверки требований;
- обработчики команд;
- отдельные решения бота.

### Integration

- цепочки событий;
- строительство и сохранение;
- отправка и возвращение флота;
- бой с применением результата;
- миграции сохранений.

### Simulation

- длительные партии по seed;
- десятки ботов;
- проверка инвариантов;
- сравнение статистики архетипов;
- поиск зависаний и бесконечных циклов.

### Smoke

- запуск приложения;
- создание партии;
- сохранение;
- production-сборка;
- загрузка ассетов на GitHub Pages.

## 10. Детерминированные тесты

Каждый тест со случайностью обязан задавать seed.

При падении длительной симуляции логируются:

- seed;
- версия правил;
- время мира;
- последнее событие;
- последние команды;
- минимальный снимок состояния.

Ошибка должна воспроизводиться локально.

## 11. Изменения схемы данных

Любое изменение сериализуемого состояния требует:

- увеличения `schemaVersion`, если формат несовместим;
- миграции;
- теста миграции;
- fixture старой версии;
- обновления документации.

Запрещено молча удалять несовместимые сохранения.

## 12. Баланс

Баланс хранится в данных.

Запрещено:

- вшивать стоимость в UI;
- дублировать формулы в нескольких местах;
- менять коэффициент без сценария проверки;
- балансировать только по одной ручной партии.

Изменение баланса должно сопровождаться:

- причиной;
- ожидаемым эффектом;
- сравнением до и после;
- набором симуляций.

## 13. Производительность

Оптимизация проводится после измерения.

Перед сложной оптимизацией нужно зафиксировать:

- сценарий;
- объём мира;
- время выполнения;
- память;
- узкое место.

Не усложнять архитектуру ради гипотетической нагрузки.

## 14. Ассеты

- все пути централизованы;
- интерфейс не зависит от размера конкретной картинки;
- отсутствующий ассет имеет безопасную заглушку;
- тяжёлые файлы оптимизируются перед добавлением;
- происхождение каждого внешнего ассета фиксируется;
- запрещено добавлять материалы с неясными правами.

## 15. Документация

Документация обновляется в том же PR, где меняется решение.

Если код расходится с документом, PR должен:

- привести код к документу;
- либо явно изменить документ и объяснить причину.

## 16. Контроль объёма

Перед добавлением новой системы нужно ответить:

1. Улучшает ли она текущий игровой цикл?
2. Есть ли зависимые механики, которые ещё не готовы?
3. Можно ли проверить её отдельно?
4. Потребует ли она поведения ботов?
5. Есть ли более простой вариант?

Если ответы неясны, система не добавляется в текущий этап.

## 17. Правило двух неудачных попыток

Если один и тот же подход дважды приводит к нерабочему решению, третья попытка должна менять подход, архитектуру или способ проверки, а не повторять тот же патч.

## 18. Минимальная проверка перед merge

```text
npm run typecheck
npm run test
npm run build
```

Позже добавляются lint, длительные simulation-тесты и browser smoke tests.

## 19. Полномочия AI-исполнителя

Владелец репозитория разрешает AI-исполнителю самостоятельно выполнять стандартный GitHub-цикл проекта без отдельного подтверждения на каждое действие:

- создавать рабочие ветки;
- создавать и изменять файлы;
- делать коммиты;
- создавать Pull Request;
- обновлять описание и состав PR;
- переводить PR из draft в ready;
- мержить PR в `main`, когда выполнены критерии готовности и доступные проверки не показывают ошибок;
- создавать последующие PR для продолжения roadmap.

### Ограничения

AI-исполнитель не должен без отдельного прямого указания:

- удалять репозиторий;
- удалять или принудительно переписывать `main`;
- использовать force-push к защищённым или общим веткам;
- отключать проверки ради прохождения merge;
- мержить PR с известными ошибками, конфликтами или повреждением данных;
- публиковать секреты, персональные данные или ассеты с неясными правами;
- менять видимость репозитория;
- создавать платные ресурсы или подключать платные внешние сервисы.

Если проверок ещё нет, допустимо мержить документацию и инфраструктурные изменения после проверки diff и отсутствия конфликтов. Рискованные или необратимые действия требуют отдельного согласования.
