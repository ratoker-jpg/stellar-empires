# Execution Roadmap Stellar Empires — PR #1–#100

**Статус:** Accepted  
**Последнее обновление:** 2026-07-19  
**Цель:** полностью реализованная одиночная браузерная космическая стратегия версии 1.0.  
**Канонический порядок:** этот документ заменяет прежний краткий план после PR #29 и является источником истины для следующих delivery batches.

## 1. Граница Release 1.0

К PR #100 игра должна поддерживать полный самостоятельный цикл:

> новая партия → выбор фракции → развитие трёх зон → исследования → производство → разведка → транспорт и экспансия → PvE/PvP против ботов → дипломатия и коалиции → борьба за стратегические звёзды → мегаструктуры → победа или поражение → итоговая статистика.

В Release 1.0 входят:

- три асимметричные фракции: Директорат «Эгида», Машинный Синод и Рой Вейра;
- одна или несколько галактик, солнечные системы, планеты, нейтральные объекты и стратегические звёзды;
- Resource, Industry и Military Zone;
- экономика, энергия, население, стабильность, склады, рынок и логистика;
- здания, исследования, корабли, оборона, очереди и офлайн-пересчёт;
- флоты, топливо, разведка, транспорт, размещение, атака, колонизация, переработка, экспедиции и специальные миссии;
- детерминированные бои, обломки, добыча, отчёты, улучшения, адмирал и командирские корабли;
- автономные честные боты, личности, дипломатия, коалиции и командная планета;
- поздняя мета-игра, редкий ресурс, промежуточные и финальная мегаструктуры;
- сохранения, импорт/экспорт, обучение, энциклопедия, достижения, настройки, доступность, арт, звук и музыка;
- массовые симуляции, балансировочный стенд, performance budget и стабильный GitHub Pages release.

Не входят в 1.0:

- сетевой мультиплеер;
- серверные аккаунты и облачные сохранения;
- монетизация, премиальная валюта и live-service;
- ручной тактический бой в реальном времени;
- обязательная сюжетная кампания.

## 2. Правила исполнения

- План содержит **100 release PR**, из них #1–#21 уже смержены, #22 фиксирует этот полный план, #23–#100 реализуют оставшиеся системы.
- Обычный delivery batch состоит из шести последовательных PR; минимум четыре сохраняется как жёсткое правило.
- Каждый PR создаётся от актуального `main`, проходит lint, typecheck, tests и production build, затем squash-merge.
- Симуляция остаётся независимой от DOM и Phaser.
- Игрок и боты используют одинаковые команды, ресурсы и ограничения.
- Каждый новый state field получает сериализацию, миграцию и тест восстановления.
- Каждый важный результат создаёт отчёт или объяснимое уведомление.
- Production-ассеты создаются только для уже работающих экранов и механик.
- Номер PR — ожидаемый GitHub-номер. При вынужденном hotfix roadmap обновляется, а оставшиеся номера не переиспользуются молча.

## 5.1 Этап B — MVP: сохранения, наука, производство и первый внешний цикл (PR #23–#34)

### PR #23 — Autosave и восстановление последней партии

Подключить IndexedDB к runtime: автосохранение после значимых команд, загрузка последнего корректного snapshot и индикатор состояния сохранения.

### PR #24 — Слоты, импорт, экспорт и аварийное восстановление

### PR #25 — Каталог исследований и дерево зависимостей

### PR #26 — Очередь исследований и реальные эффекты

Сериализуемая очередь, стоимость, время, события завершения и эффекты для экономики, строительства, разведки и кораблей.

### PR #27 — Схемы кораблей, обороны и планетарный инвентарь

Единые валидируемые определения юнитов, планетарные запасы кораблей/обороны, население, ангар и требования.

### PR #28 — Верфь и оборонное производство

Очереди Shipyard/Defense, пакетное производство, отмена, офлайн-завершение и первые рабочие корабли/турели.

### PR #29 — Флот как сериализуемая сущность

Состав, груз, скорость, вместимость, полётные слоты, резервирование кораблей и запрет дублирования.

### PR #30 — Расстояние, топливо и жизненный цикл полёта

Единая формула расстояния, топливо, отправка, прибытие, удержание, отзыв, возврат и восстановление после закрытия страницы.

Миссии transport/deploy, загрузка ресурсов, разгрузка, гарнизоны, потеря цели и конфликт одновременных прибытий.

### PR #32 — Разведка и контрразведка

Уровни разведки, объём раскрываемых данных, обнаружение, устаревание сведений, память наблюдений и разведывательные отчёты.

Группы юнитов, приоритеты целей, раунды, урон, защита, победа/ничья/отступление и воспроизводимый battle seed.

### PR #34 — Обломки, добыча и боевые отчёты

Потери, захват ресурсов, поле обломков, переработка, подробный лог раундов и ссылки отчёта на цели.

**Gate после #34:** партия переживает перезапуск, игрок исследует технологию, производит корабли, отправляет флот, разведывает и проводит первый бой с полноценным отчётом.

## 6. Сквозные критерии готовности каждого PR

Каждый implementation PR считается готовым только когда:

1. затронутые типы и данные валидируются;
2. новая команда имеет успешные и ошибочные тесты;
3. событие выполняется ровно один раз;
4. сохранение переживает round-trip и миграцию;
5. бот не получает недоступные игроку данные;
6. UI объясняет блокировку или результат;
7. нет необработанного `Math.random`, системного времени или DOM-зависимости внутри simulation core;
8. CI проходит lint, typecheck, Vitest и production build;
9. Pages не получает отсутствующие файлы или неверный base path;
10. roadmap и документация обновляются, если фактический scope изменился.

## 9. Решения, которые уточняются прототипами

До соответствующего PR нельзя считать финальными:

- длительность партии и скорость игровых часов;
- размер мира и количество одновременно активных ботов;
- окончательная экономика энергии, населения и стабильности;
- рыночные комиссии и ограничения;
- формула топлива и количество полётных слотов;
- окончательная контр-система оружия и защиты;
- число раундов боя и правила отступления;
- сила адмирала и командирских кораблей;
- условия вступления в коалиции;
- темп получения мета-ресурса;
- длительность уязвимости финального проекта;
- баланс асимметрии трёх фракций.

Решения принимаются через собственные тесты и массовые прогоны. Исторические числа Nemexia используются только как исследовательский ориентир и не копируются автоматически.
