# Simulation and Data

**Статус:** Draft  
**Последнее обновление:** 2026-07-18

## 1. Цель

Симуляция должна корректно работать:

- в реальном времени;
- при ускорении времени;
- после закрытия страницы;
- при массовом развитии ботов;
- в автоматических тестах;
- после обновления версии игры.

## 2. Игровое время

Игра использует собственное игровое время в миллисекундах Unix UTC.

Основные значения:

- `savedAt` — когда файл был записан;
- `simulatedUntil` — до какого момента мир фактически рассчитан;
- `targetTime` — до какого момента нужно продвинуть симуляцию;
- `speedMultiplier` — выбранная скорость активной игры.

Визуальные часы не являются источником истины.

## 3. Событийная очередь

Вместо постоянного пересчёта всего мира используется приоритетная очередь событий.

Примеры:

- завершение здания;
- завершение исследования;
- завершение партии кораблей;
- прибытие флота;
- возвращение флота;
- завершение колонизации;
- плановый ход бота;
- истечение временного эффекта.

```ts
interface ScheduledEvent {
  id: string;
  type: string;
  executeAt: number;
  sequence: number;
  payload: unknown;
}
```

Сортировка выполняется по `executeAt`, затем по `sequence`. Это сохраняет однозначный порядок событий с одинаковым временем.

## 4. Продвижение симуляции

Алгоритм:

1. Загрузить состояние.
2. Определить `targetTime`.
3. Взять ближайшее событие.
4. Если событие позже `targetTime`, остановиться.
5. Обновить непрерывные показатели до времени события.
6. Выполнить событие.
7. Создать новые события при необходимости.
8. Повторять до достижения `targetTime`.
9. Обновить непрерывные показатели до `targetTime`.
10. Записать `simulatedUntil`.

## 5. Непрерывная добыча

Ресурсы не обновляются каждую секунду.

При чтении или событии рассчитывается прирост за интервал:

```text
newAmount = min(storageCapacity, oldAmount + productionPerSecond × elapsedSeconds)
```

Если в интервале меняется производство, интервал делится событием завершения здания или изменения энергии.

## 6. Детерминизм

Одинаковые:

- начальное состояние;
- seed;
- порядок команд;
- версия правил;

должны давать одинаковый результат.

Для случайных решений используется собственный генератор псевдослучайных чисел с сериализуемым состоянием. Нельзя использовать `Math.random()` в доменной логике.

## 7. Идентификаторы

Игровые сущности используют стабильные строковые идентификаторы:

- `empireId`;
- `planetId`;
- `fleetId`;
- `eventId`;
- `reportId`.

Идентификаторы контента фиксированы в конфигурации:

- `metal_mine`;
- `energy_reactor`;
- `light_fighter`;
- `espionage_tech`.

Отображаемые названия не используются как ключи.

## 8. Состояние мира

Предварительная структура:

```ts
interface WorldState {
  meta: WorldMeta;
  playerEmpireId: string;
  empires: Record<string, EmpireState>;
  planets: Record<string, PlanetState>;
  fleets: Record<string, FleetState>;
  debrisFields: Record<string, DebrisFieldState>;
  reports: Record<string, ReportState>;
  diplomacy: DiplomacyState;
  rng: RandomState;
}
```

Данные не должны содержать экземпляры классов, Phaser-объекты, DOM-узлы или функции.

## 9. Конфигурация контента

Статические правила отделяются от состояния сохранения.

Категории данных:

- здания;
- исследования;
- корабли;
- оборона;
- миссии;
- формулы;
- настройки сложности;
- архетипы ботов.

Каждая запись проходит runtime-валидацию при запуске разработки и тестов.

## 10. Формулы

Формулы стоимости и времени должны быть централизованы.

Пример общего подхода:

```text
cost(level) = baseCost × growthFactor^(level - 1)
time(level) = baseTime × timeFactor^(level - 1) / speedModifiers
```

Конкретные коэффициенты не фиксируются до появления первого балансировочного набора.

## 11. IndexedDB

Хранилища первой версии:

- `saveSlots` — метаданные слотов;
- `snapshots` — полные состояния;
- `settings` — пользовательские настройки;
- `crashBackups` — аварийные копии.

Запись должна быть транзакционной. Новый snapshot сначала полностью сохраняется, затем становится активным.

## 12. Версионирование сохранений

У сохранения есть `schemaVersion`.

Для каждого изменения схемы создаётся миграция:

```text
v1 → v2 → v3
```

Правила:

- миграции не изменяют оригинальный экспорт без резервной копии;
- миграция либо завершается полностью, либо откатывается;
- несовместимая версия показывает понятную ошибку;
- тестовые сохранения прошлых версий хранятся в fixtures.

## 13. Экспорт и импорт

Экспорт содержит:

- метаданные;
- состояние мира;
- очередь событий;
- состояние RNG;
- контрольную сумму;
- версию схемы и игры.

Импорт проверяет:

- корректный JSON;
- допустимую структуру;
- размер файла;
- контрольную сумму;
- поддержку версии;
- отсутствие опасных полей.

## 14. Отчёты

Отчёт — неизменяемый результат произошедшего события.

Он может содержать:

- участников;
- время;
- известные игроку входные данные;
- итог;
- потери;
- добычу;
- ссылку на связанные события;
- версию боевого движка.

Старые подробные отчёты могут архивироваться или агрегироваться для ограничения размера сохранения.

## 15. Офлайн-пересчёт

При долгом отсутствии нельзя блокировать интерфейс одним огромным циклом.

Подход:

- расчёт выполняется в Worker;
- события обрабатываются пакетами;
- Worker периодически сообщает прогресс;
- пользователь видит диапазон пересчитываемого времени;
- при критической ошибке исходное сохранение остаётся доступным.

## 16. Ограничение catch-up

На первом этапе максимальный офлайн-интервал может быть ограничен, например 30 игровыми днями. Это защитит браузер от неконтролируемого пересчёта до оптимизации.

Ограничение должно быть явным и настраиваемым, а не скрытым.

## 17. Инварианты

Симуляция обязана сохранять:

- ресурсы не ниже нуля;
- количество кораблей — целое и неотрицательное;
- одна сущность принадлежит существующему владельцу;
- флот не находится одновременно на планете и в полёте;
- завершённое событие не выполняется повторно;
- занятая позиция галактики уникальна;
- очередь не содержит ссылок на удалённые сущности без допустимой причины.

Инварианты проверяются в тестах и в development-сборке.
