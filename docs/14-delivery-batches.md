# Delivery Batches

**Статус:** Accepted  
**Последнее обновление:** 2026-07-18

## 1. Назначение

Разработка Stellar Empires ведётся последовательными пакетами поставки — **delivery batches**.

Обычный пакет включает минимум **4 самостоятельных Pull Request**, объединённых одной целью roadmap.

## 2. Правило пакета

Внутри одного рабочего цикла AI-исполнитель:

1. определяет 4–6 связанных, но независимо проверяемых задач;
2. выполняет первую задачу в отдельной ветке;
3. создаёт Pull Request;
4. проверяет diff и доступные автоматические проверки;
5. самостоятельно исправляет найденные ошибки;
6. мержит зелёный PR в `main`;
7. создаёт следующую ветку уже от обновлённого `main`;
8. продолжает до завершения минимум четырёх PR либо до реального блокера.

PR внутри пакета выполняются **последовательно**, а не параллельно. Это исключает накопление конфликтов и позволяет каждой следующей задаче опираться на уже проверенный код.

## 3. Когда пакет можно остановить раньше

Пакет допускается остановить до четырёх PR только при блокере, который нельзя безопасно устранить внутри проекта:

- GitHub или CI недоступен;
- требуются секреты или платная инфраструктура;
- нужна лицензия или право на внешний материал;
- найден риск повреждения пользовательских данных;
- необходимое действие запрещено правилами безопасности проекта.

Обычная ошибка кода, теста, типов или сборки блокером не считается: AI-исполнитель обязан исправить её самостоятельно.

## 4. Размер PR

Каждый PR должен:

- иметь одну проверяемую цель;
- оставлять `main` в рабочем состоянии;
- включать тесты для новой доменной логики;
- обновлять документацию при изменении архитектуры или формата данных;
- не смешивать независимые крупные системы;
- содержать описание проверки, рисков и намеренно отложенной работы.

## 5. Проверка и merge

Минимальный путь:

```text
branch → diff review → PR → CI → исправление при необходимости → squash merge → next branch
```

Запрещено открывать следующий зависимый PR от незамерженной ветки, если нет конкретной причины использовать stacked PR.

## 6. Планирование пакета

Перед началом пакета фиксируются:

- цель пакета;
- список PR в порядке выполнения;
- зависимость каждого PR;
- критерий завершения пакета;
- известные риски.

План может меняться после результатов CI или обнаружения архитектурной проблемы, но количество PR не уменьшается без реального блокера: вместо отпавшей задачи выбирается следующая безопасная задача roadmap.

## 7. Текущий пакет B1

Цель: перейти от технического каркаса к воспроизводимому игровому миру.

1. **B1.1 — Delivery workflow:** зафиксировать пакетный режим работы.
2. **B1.2 — Simulation core:** команды, события, очередь, часы, reducer и replay.
3. **B1.3 — Persistence core:** схема сохранения, адаптер хранения, экспорт и импорт.
4. **B1.4 — Galaxy model:** детерминированная генерация систем и отображение модели в Phaser.

Пакет завершён, когда все четыре PR смержены, CI зелёный, а новая партия создаёт воспроизводимую галактику и может быть сериализована без зависимости от Phaser.
