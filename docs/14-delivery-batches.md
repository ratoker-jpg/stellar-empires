# Delivery Batches

**Статус:** Accepted  
**Последнее обновление:** 2026-07-19

## 1. Назначение

Разработка Stellar Empires ведётся последовательными пакетами поставки — **delivery batches**.

Стандартный пакет включает **6 самостоятельных Pull Request**, объединённых одной целью roadmap.

AI-исполнитель может самостоятельно расширить пакет до **7 или 8 PR**, когда дополнительные задачи являются прямым безопасным продолжением того же этапа.

## 2. Стандартный пакет

Внутри одного рабочего цикла AI-исполнитель:

1. определяет шесть связанных, но независимо проверяемых задач;
2. выполняет первую задачу в отдельной ветке;
3. создаёт Pull Request;
4. проверяет diff и доступные автоматические проверки;
5. самостоятельно исправляет найденные ошибки;
6. мержит зелёный PR в `main`;
7. создаёт следующую ветку уже от обновлённого `main`;
8. продолжает до завершения шести PR либо до реального блокера;
9. обновляет статус и handoff-документы.

PR внутри пакета выполняются **последовательно**, а не параллельно. Это исключает накопление конфликтов и позволяет каждой следующей задаче опираться на уже проверенный код.

Достижение четырёх или пяти PR само по себе больше не является основанием завершить обычный пакет.

## 3. Когда пакет расширяется до 7–8 PR

AI-исполнитель не обязан отдельно спрашивать владельца и может продолжить после шестого PR, когда одновременно выполнены условия:

- следующие один или два пункта уже зафиксированы roadmap;
- они непосредственно продолжают результат первых шести PR;
- зависимости понятны и уже доступны в `main`;
- CI стабилен;
- каждый PR остаётся небольшим и самостоятельно проверяемым;
- нет нерешённой архитектурной развилки;
- нет риска потери данных, проблем лицензирования, секретов или платной инфраструктуры;
- завершение восьмого PR создаёт более цельный продуктовый результат и уменьшает стоимость следующего handoff.

Расширение до восьми не должно использоваться для объединения несвязанных систем или для скрытого увеличения размера отдельных PR.

## 4. Когда пакет можно остановить раньше

Пакет допускается остановить до шести PR только если:

- текущий roadmap-этап объективно заканчивается меньшим числом задач;
- владелец явно запросил меньший пакет;
- обнаружен блокер, который нельзя безопасно устранить внутри проекта.

Реальные блокеры:

- GitHub или CI недоступен;
- требуются отсутствующие секреты или разрешения;
- требуется платная инфраструктура;
- нужна лицензия или право на внешний материал;
- найден риск повреждения пользовательских данных;
- необходимое действие запрещено правилами безопасности проекта.

Обычная ошибка кода, теста, типов, lint или сборки блокером не считается: AI-исполнитель обязан исправить её самостоятельно.

## 5. Размер PR

Каждый PR должен:

- иметь одну проверяемую цель;
- оставлять `main` в рабочем состоянии;
- включать тесты для новой доменной логики;
- обновлять документацию при изменении архитектуры или формата данных;
- не смешивать независимые крупные системы;
- содержать описание проверки, рисков и намеренно отложенной работы.

Количество PR в пакете не оправдывает искусственное дробление пустых или формальных изменений. Каждый PR должен давать отдельный проверяемый результат.

## 6. Проверка и merge

Минимальный путь:

```text
branch → diff review → PR → CI → исправление при необходимости → squash merge → next branch from updated main
```

Запрещено открывать следующий зависимый PR от незамерженной ветки, если нет конкретной документированной причины использовать stacked PR.

Для каждого PR стандартная проверка включает:

```text
lint → typecheck → tests → production build
```

Если добавленная система требует миграции, simulation-прогона, browser smoke test или проверки ассетов, эти проверки добавляются сверх стандартного CI.

## 7. Планирование пакета

Перед началом пакета фиксируются:

- цель пакета;
- шесть стандартных PR в порядке выполнения;
- возможные седьмой и восьмой PR;
- зависимость каждого PR;
- критерий завершения пакета;
- известные риски.

После шестого PR исполнитель оценивает расширение до семи или восьми по критериям раздела 3. Спрашивать владельца для безопасного расширения не требуется.

План может меняться после результатов CI или обнаружения архитектурной проблемы. Отпавшая задача заменяется следующей безопасной задачей roadmap, если это не нарушает целостность пакета.

## 8. Протокол нового чата

В начале нового AI-сеанса исполнитель обязан прочитать:

1. `AGENTS.md`;
2. `docs/17-continuation-guide.md`;
3. `docs/project-status.json`;
4. `docs/16-execution-roadmap.md`;
5. `docs/roadmap-pr-index.json`;
6. актуальные последние PR в GitHub.

Нельзя опираться только на пересказ пользователя или память предыдущего чата. Источником истины являются `main`, GitHub history и перечисленные документы.

## 9. Завершение и handoff

После каждого пакета нужно зафиксировать:

- номера и merge SHA всех PR;
- что теперь реально работает;
- какие ограничения остались;
- какой PR идёт следующим;
- следующие шесть стандартных PR;
- какие седьмой и восьмой PR можно безопасно добавить;
- результаты CI и известных дополнительных проверок.

Статус хранится в `docs/project-status.json`, а правила продолжения — в `docs/17-continuation-guide.md`.
