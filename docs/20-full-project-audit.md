## 3. Итоговая оценка

### Главный вывод

**Stellar Empires уже является сильным техническим прототипом, но ещё не является цельной законченной игрой.**

Проект заметно опережает обычный любительский прототип по инженерной дисциплине:

- детерминированная симуляция;
- строгие TypeScript-контракты;
- миграции сохранений;
- единые команды для игрока и ботов;
- честная модель восприятия AI;
- автоматические проверки;
- отдельная design system;
- нормальная документация и последовательные PR.

Основной риск сейчас не в недостатке механик. Механик уже много. Риск в том, что проект слишком быстро развивается **в ширину**, а ключевые вопросы реальной партии остаются без проверки:

- корректно ли живут боты при ускорении времени;
- интересно ли играть полный цикл от старта до победы;
- где находится реальная сложность;
- не ломается ли экономика в середине партии;
- не разрастаются ли сохранения и runtime state;
- понимает ли игрок, что ему делать и зачем;
- выдерживает ли игра длинную сессию;
- одинаково ли воспроизводится партия после save/load.

### Оценка по областям

| Область | Оценка | Комментарий |
|---|---:|---|
| Архитектура simulation | 8/10 | Правильное разделение доменов и UI, но reducer и state начинают разрастаться |

- **как инженерный прототип:** примерно 6–7/10;

- **как законченная PvE-игра:** примерно 3–4/10.

## 4. Как проект работает сейчас

## 4.1. Технологический стек

Проект использует:

- Phaser 4.2.1;
- TypeScript 6;
- Vite 8;
- Vitest 4;
- ESLint 10;
- IndexedDB;
- Web Worker для планирования ботов;
- GitHub Actions;
- GitHub Pages.

Phaser используется для интерактивной карты галактики. Большинство управленческих экранов реализованы через обычный HTML, DOM и CSS.

Это правильный гибрид для браузерной стратегии:

- canvas отвечает за пространственную сцену;
- DOM отвечает за формы, списки, карточки, таблицы, диалоги и accessibility.

Полный перенос UI в Phaser не требуется. Переписывание проекта на React также не требуется.

## 4.2. Запуск приложения

`src/main.ts` выполняет роль composition root:

1. подключает CSS и faction assets;
2. загружает autosave;
3. при отсутствии сохранения создаёт новую игру;
4. создаёт Phaser instance;
5. монтирует планетарный экран;
6. создаёт command bridge;
7. монтирует остальные экраны;
8. запускает autosave;
9. запускает автономный bot scheduler.

Бизнес-логика в основном не находится внутри `main.ts`, что является правильным решением. Однако количество импортов и mount-вызовов уже показывает, что runtime composition необходимо постепенно переводить на registry или application controller.

## 4.3. Каноническое состояние

`GameState` schema v13 хранит:

- seed и игровое время;
- список империй;
- галактику;
- колонии и нейтральные планеты;
- исследования;
- улучшения кораблей;
- командиров и doctrine progression;
- флоты;
- разведку;
- поля обломков;
- логистические маршруты;
- рынок;
- космические объекты;
- стратегические ресурсы;
- мировые события;
- pending events;
- command log;
- event log.

Это уже полноценный центральный state большой стратегии. Его дальнейшее расширение дипломатией, коалициями и endgame возможно, но перед этим необходимо решить вопросы размера state, журналов и временной симуляции.

## 4.4. Команды

Игрок и боты используют одни и те же `GameCommand`:

- управление временем;
- строительство;
- специализация планеты;
- логистика;
- рынок;
- исследования;
- производство;
- ремонт обороны;
- улучшение кораблей;
- формирование флота;
- doctrine и flagship;
- отправка и отзыв флота;
- экспедиции;
- операции на космических объектах.

Команда проходит через `executeCommand`, который возвращает:

- успешное новое состояние;
- либо `code`, `message` и дополнительные детали ошибки.

Это сильный архитектурный контракт. Он позволяет:

- применять одинаковые правила к игроку и AI;
- тестировать домены отдельно;
- логировать причины отказа;
- воспроизводить действия;
- строить headless simulation.

## 4.5. Игровое время

Сейчас время является дискретным и управляется игроком:

- `+1 мин`;
- `+10 мин`;
- `+1 час`;
- переход до следующего события.

`ADVANCE_TIME` последовательно:

1. начисляет ресурсы;
2. обрабатывает scheduled events;
3. обрабатывает логистику;
4. оценивает мировые события;
5. устанавливает итоговое время.

Проблема: решения ботов не встроены в этот event loop и выполняются отдельным runtime scheduler после изменения state. Это создаёт критическую рассинхронизацию, подробно описанную ниже.

## 4.6. Новая партия

Новая партия создаётся через фиксированный seed source `stellar-empires-m1`. Поэтому текущие новые игры фактически используют одну и ту же детерминированную галактику.

На старте создаются:

- игрок;
- Aegis bot;
- Synod bot;
- Veyra bot;
- домашние колонии;
- нейтральные пиратские силы;
- космические объекты;
- мировой event state;
- рынок;
- research, ship upgrades и command states.

Текущий старт подходит для повторяемого тестового сценария, но не подходит как финальная new game flow.

## 4.7. Основной игровой цикл

Текущий цикл выглядит так:

1. игрок выбирает фракцию;
2. развивает домашнюю колонию;
3. строит ресурсные, промышленные и военные здания;
4. исследует технологии;
5. производит корабли и оборону;
6. формирует флоты;
7. разведывает галактику;
8. атакует ботов или пиратов;
9. собирает обломки и добычу;
10. колонизирует новые планеты;
11. настраивает логистику;
12. использует рынок;
13. участвует в экспедициях и операциях на объектах;
14. реагирует на мировые события;
15. улучшает корпуса, doctrine и flagship.

Цикл уже широкий, но пока не замыкается в ясную победу или поражение.

## 5. Что сделано хорошо

## 5.1. Simulation отделена от UI

Это одно из главных достоинств проекта.

Доменная логика не завязана напрямую на:

- DOM;
- конкретные кнопки;
- Phaser scene;
- системные таймеры браузера.

Такой подход позволяет:

- запускать бои и экономику без интерфейса;
- писать детерминированные тесты;
- строить массовые симуляции;
- безопаснее мигрировать сохранения;
- заменять UI без переписывания ядра.

Это решение необходимо сохранить.

## 5.2. Детерминированная модель

Проект осознанно избегает случайности через `Math.random()` и системное время внутри simulation. События завязаны на seed, state и event sequence.

Плюсы:

- воспроизводимые тесты;
- стабильные save/load;
- возможность replay;
- сравнимые balance runs;
- диагностика редких ошибок.

Это особенно важно для одиночной стратегии с большим числом взаимосвязанных систем.

## 5.3. Игрок и боты используют одинаковые правила

AI не получает отдельные скрытые команды или прямое изменение ресурсов. Bot planners формируют обычные `GameCommand`, а reducer повторно валидирует их.

Это создаёт честный фундамент PvE:

- AI не обходит требования зданий;
- AI не строит без ресурсов;
- AI не атакует через невозможную миссию;
- AI можно отлаживать теми же reason codes, что и действия игрока.

## 5.4. Честное восприятие ботов

Боты имеют отдельную perception/memory model:

- свои планеты и флоты известны полностью;
- чужие данные доступны через observations;
- устаревшая разведка хранится как stale memory;
- скрытые изменения игрока не должны утекать в bot planner.

Это правильнее, чем делать AI «умнее» за счёт полного доступа к state.

## 5.5. Хорошая persistence-база

Реализованы:

- IndexedDB autosave;
- отдельные save slots;
- import/export;
- checksum;
- recovery snapshot;
- schema migrations;
- runtime validation загруженного состояния.

Для длинной стратегии это критически важный слой. Потеря сохранения наносит игре больший ущерб, чем отдельная ошибка интерфейса.

## 5.6. Data-driven каталоги

Здания, исследования, юниты, combat profiles и faction mechanical IDs постепенно переводятся в каталоги и registry.

PR #76 и #77 создают правильное направление:

- стабильные ID;
- faction namespace;
- dependency validation;
- явная политика legacy alias;
- подготовка к механически разным фракциям.

Это правильный фундамент для Synod и Veyra.

## 5.7. PvE уже не ограничивается одним типом боя

Есть:

- нейтральные пиратские базы;
- экспедиции;
- астероиды;
- газовые облака;
- аномалии;
- мировые события;
- anti-farm diminishing returns;
- threat scaling;
- unified reports.

Это даёт потенциал для живой одиночной галактики, а не только для прямых атак на трёх ботов.

## 5.8. Визуальная система имеет единые правила

В проекте уже существуют:

- design tokens;
- faction colors;
- panel primitives;
- кнопки и состояния;
- отступы и размеры;
- focus-visible;
- reduced motion;
- high contrast;
- compact и mobile режимы.

Визуальное развитие больше не является набором несвязанных экранов.

## 5.9. CI и delivery дисциплинированы

Каждый PR проверяется через:

1. lint;
2. TypeScript typecheck;
3. Vitest;
4. production build.

Есть отдельный deploy workflow на GitHub Pages. PR имеют ограниченный scope, описание границ и тестов.

Это хороший процесс для проекта, который часто развивается через AI-агентов.

## 6. Критические проблемы P0

## P0-1. Ускорение времени ломает развитие ботов

Это главный текущий дефект игровой модели.

### Как работает сейчас

Bot profile задаёт `decisionIntervalSeconds`:

- Aegis: 600 секунд;
- Synod: 300 секунд;
- Veyra: 450 секунд.

Bot scheduler проверяет, прошло ли достаточно времени с последнего решения. Если прошло — выполняет один decision batch и записывает cursor равным текущему игровому времени.

### Почему это неправильно

Игрок может выполнить `ADVANCE_TIME` сразу на 3600 секунд.

Логически за этот час:

- Aegis должен получить до 6 decision windows;
- Synod — до 12;
- Veyra — до 8.

Фактически каждый бот получает один цикл решений после завершения большого скачка.

### Последствия

- игрок развивается быстрее AI;
- сложность зависит от того, насколько крупными шагами игрок двигает время;
- автоматические тесты отдельного planner могут проходить, но реальная партия остаётся нечестной;
- balance values не имеют смысла, пока cadence реализован неверно;
- offline resimulation в будущем будет давать ещё более сильный перекос.

### Что нужно сделать

Bot decisions должны стать каноническими simulation events.

Рекомендуемая модель:

- хранить `nextDecisionAt` по каждой bot empire внутри `GameState`;
- включить bot decision time в `earliestTime()` внутри `ADVANCE_TIME`;
- на каждом наступившем decision point выполнять один допустимый batch;
- последовательно догонять все пропущенные интервалы;
- ограничить максимум итераций на один вызов для защиты от зависания;
- сериализовать bot timing в save;
- тестировать эквивалентность `+60 × 60` и `+3600`.

### Критерий приёмки

Одинаковый seed и одинаковая последовательность команд игрока должны давать одинаковый state независимо от размера шагов времени.

## P0-2. Save/load меняет поведение AI

Сейчас bot cursor хранится в `BotAutomationController`, а не в `GameState`.

После загрузки controller создаёт новый cursor, позволяющий немедленное решение. Поэтому save/load может добавить ход бота, которого не было бы при непрерывной игре.

### Последствия

- нарушается полная детерминированность;
- replay и checksum не отражают runtime AI timing;
- загрузка может менять баланс;
- трудно воспроизводить баги после сохранения.

### Решение

Bot scheduler cursor или `nextDecisionAt` должен быть частью schema и migration.

## P0-3. Нет завершённой полной партии

Сейчас в игре нет готового условия победы и поражения.

Отсутствуют:

- финальная цель;
- проигрыш;
- post-round экран;
- ясное завершение гонки;
- проверенный endgame pacing.

Roadmap планирует дипломатию, коалиции, strategic stars и transit nexus позже. Однако ждать полного большого endgame перед тестированием полной партии рискованно.

### Что нужно сделать раньше

Добавить временный вертикальный endgame:

1. уничтожить или исследовать ключевые PvE-цели;
2. открыть финальную технологию;
3. захватить стратегический объект;
4. выдержать финальную волну;
5. получить победу или поражение;
6. показать статистику партии.

Этот endgame может быть позже заменён coalition/transit nexus системой, но уже сейчас позволит проверить:

- длину партии;
- темп экономики;
- силу ботов;
- полезность исследований;
- частоту боёв;
- понятность целей.

## P0-4. Все новые партии фактически одинаковые

Фиксированный seed удобен для разработки, но не должен быть единственным production flow.

### Необходимо добавить

- случайный seed по умолчанию;
- ручной ввод seed;
- копирование seed;
- выбор размера галактики;
- количество AI;
- уровень сложности;
- длину партии;
- режим тестового фиксированного seed.

## P0-5. State и сохранения будут бесконечно расти

`commandLog` и `eventLog` находятся внутри `GameState` и сохраняются полностью.

Каждая команда:

- добавляется в immutable array;
- сериализуется в IndexedDB;
- участвует в checksum;
- копируется в Web Worker вместе со всем state.

### Риск длинной партии

- растущий save size;
- более медленный autosave;
- дорогой `JSON.stringify`;
- долгий structured clone;
- рост памяти;
- задержки bot scheduler;
- O(n) копирование журналов на каждую команду.

### Рекомендуемая модель

Разделить:

1. **канонический snapshot** — текущее состояние мира;
2. **ограниченный audit tail** — последние N команд;
3. **агрегированные reports** — необходимые игроку данные;
4. **опциональные replay segments** — отдельное хранилище для отладки;
5. **UI notification history** — не часть simulation state.

Необходимо установить budgets:

- максимальный размер autosave;
- максимальная длина command log;
- максимальная длина event history;
- максимальное время serialize/parse;
- максимальный Worker payload.

## 7. Игровой дизайн и баланс

## 7.1. Что уже хорошо

Игра имеет достаточно слоёв для интересной стратегии:

- развитие колонии;
- ограниченные поля зон;
- энергия;
- население;
- стабильность;
- складские лимиты;
- research dependencies;
- производственные очереди;
- несколько типов миссий;
- разведка;
- counters в бою;
- формации;
- target priorities;
- command doctrine;
- flagship;
- рынок и логистика;
- PvE и мировые события.

Основная проблема не в количестве механик, а в отсутствии доказательства, что они формируют хорошие решения в полной партии.

## 7.2. Экономика требует раннего simulation harness

Стоимость зданий масштабируется примерно на 60% за уровень, время — примерно на 45%, а многие производственные эффекты растут линейно.

Без массовых прогонов возможны:

- слишком быстрый старт;
- резкий midgame wall;
- бессмысленные высокие уровни;
- один очевидный build order;
- чрезмерная важность энергии;
- отсутствие реального выбора между колониями;
- экономический snowball после первой успешной атаки.

Balance harness запланирован слишком поздно. Его нужно начать до дипломатии и endgame.

### Минимальный harness должен измерять

- время до первой новой технологии;
- время до первого флота;
- время до первой колонии;
- время до первого боя;
- среднее число простаивающих очередей;
- средний ресурсный overflow;
- частоту энергетического дефицита;
- эффективность каждого здания;
- стоимость одного процента военной силы;
- время восстановления после поражения;
- долю партий, где один AI выходит в необратимый snowball.

## 7.3. Не хватает слоя стратегических намерений

Игрок видит много локальных действий, но не всегда видит верхнеуровневую цель.

Интерфейс должен отвечать:

- что сейчас является главной угрозой;
- какой следующий unlock наиболее важен;
- почему остановлено производство;
- где узкое место империи;
- что приближает победу;
- насколько игрок отстаёт от AI;
- какие события требуют реакции.

Нужен единый **операционный журнал решений**, а не просто ещё один экран со статистикой.

## 7.4. Экзотическая материя пока является тупиковым ресурсом

Редкий ресурс добывается, но не имеет полноценного расходного цикла.

До endgame необходимо добавить хотя бы промежуточные траты:

- ускоренное восстановление;
- временные fleet modules;
- reroll expedition outcome preview;
- усиление strategic scan;
- activation cost редких объектов;
- промежуточные anchor arrays.

Награда без выбора расходования быстро теряет ценность.

## 7.5. Фракционная асимметрия должна менять решения, а не только цифры

После #77 Aegis получает полный нативный каталог. Synod и Veyra должны не просто иметь другие коэффициенты.

Хорошая асимметрия должна менять:

- оптимальный порядок развития;
- отношение к колониям;
- риск флотов;
- способы разведки;
- восстановление;
- экономические ограничения;
- полезность doctrine;
- подход к PvE;
- реакцию на мировые события.

Пример:

- Aegis — предсказуемая инфраструктура, оборона и логистика;
- Synod — мобильное исследование, адаптация и использование объектов;
- Veyra — давление, быстрые ударные окна и рискованные преобразования ресурсов.

## 8. Боты и PvE

## 8.1. Сильные стороны

Текущий AI имеет хорошую базу:

- личности;
- difficulty profiles;
- economy planner;
- research/production planner;
- fleet planner;
- threat/recovery planner;
- честную perception model;
- reason codes;
- повторную проверку команд reducer.

## 8.2. Главная архитектурная проблема

AI сейчас планирует короткий набор ближайших допустимых команд, но не хранит долгосрочное намерение.

Это может приводить к:

- частой смене приоритета;
- циклическому поведению;
- локально разумным, но глобально бессмысленным действиям;
- отсутствию подготовки сложных операций;
- неспособности копить ресурсы ради крупной цели.

## 8.3. Что нужно добавить

Ввести слой `StrategicIntent`:

```text
Цель: захватить систему Helion 04
Этапы:
1. поднять sensor coverage;
2. построить разведчик;
3. получить разведданные уровня 3;
4. накопить топливо;
5. собрать атакующий флот;
6. атаковать при приемлемом соотношении сил.
```

Intent должен хранить:

- цель;
- причину выбора;
- текущий этап;
- ожидаемую стоимость;
- timeout;
- условия отмены;
- cooldown на повторный выбор.

## 8.4. Difficulty не должна быть только cadence

Сложность лучше формировать через качество решений, а не скрытые бонусы.

Возможные уровни:

- easy: слабее оценивает угрозу, чаще меняет цель, не оптимизирует логистику;
- normal: использует стандартный planner;
- hard: лучше прогнозирует ресурсы, координирует несколько шагов и сохраняет резервы.

При этом все уровни должны использовать одинаковые правила и ресурсы.

## 8.5. PvE должно давать выбор, а не только reward roll

Экспедиции и события следует развивать через решения:

- рискнуть кораблями ради большей награды;
- потратить ресурс на безопасный маршрут;
- продолжить исследование или уйти;
- помочь одной стороне события;
- временно изменить экономическую специализацию;
- выбрать между немедленной добычей и долгосрочным control bonus.

## 9. Визуал и UX

## 9.1. Что хорошо

Проект имеет узнаваемое направление:

- тёмный industrial sci-fi;
- cyan/gold акценты;
- отдельные faction palettes;
- крупный HUD;
- постоянная навигация;
- центральная игровая сцена;
- контекстные панели;
- единые cards и buttons.

## 9.2. Основная проблема: интерфейс выглядит как административный dashboard

На экранах много:

- рамок;
- одинаковых карточек;
- мелких подписей;
- числовых метрик;
- вторичной информации;
- технических терминов.

В результате теряется ощущение управления космической империей.

### Как улучшить

На каждом экране определить:

1. один главный объект;
2. одно главное решение;
3. одно главное предупреждение;
4. вторичные данные, скрываемые по запросу.

Пример для планеты:

- главный объект — визуальная колония или выбранное здание;
- основное действие — построить/улучшить;
- предупреждение — энергия или заполнение зоны;
- детали — requirements, точные коэффициенты, secondary metrics.

## 9.3. Фракции визуально различаются недостаточно глубоко

Сейчас большое различие строится на accent color.

Следующие слои идентичности:

- разная геометрия панелей;
- разные разделители и рамки;
- разная плотность интерфейса;
- собственная иконографика;
- разные motion patterns;
- разные способы отображать статус;
- faction-specific wording;
- отдельная звуковая среда.

## 9.4. В production shell остаются следы прототипа

В `index.html` остаются:

- disabled navigation buttons, которые позже активируются модулями;
- hidden showcase containers;
- asset preview dialogs;
- англоязычные служебные подписи;
- тестовые названия atlas.

Это создаёт:

- хрупкий DOM contract;
- риск случайного появления dev UI;
- смешение production и sandbox;
- сложность рефакторинга навигации.

### Решение

Создать screen registry:

```ts
interface ScreenDefinition {
  id: string;
  label: string;
  icon: string;
  mount: () => ScreenController;
  isAvailable: (state: GameState) => boolean;
}
```

Навигация должна строиться из registry, а не патчиться отдельными mount-функциями.

## 9.5. Accessibility

Сильные части:

- skip link;
- live region;
- focus-visible;
- Escape для dialogs;
- keyboard navigation в HUD;
- reduced motion;
- prefers contrast;
- mobile layouts.

Проблема: интерактивные объекты Phaser map в основном pointer-driven.

Необходимо гарантировать полноценный DOM-эквивалент:

- список систем;
- поиск;
- фильтры;
- выбор цели;
- отправка миссии;
- доступ к информации без canvas.

Accessibility считается закрытой только если все игровые действия можно выполнить без мыши и без чтения canvas.

## 9.6. Onboarding

Сейчас пользователь получает много систем почти одновременно.

Нужен сценарий первых 20–30 минут:

1. построить первый объект;
2. понять энергию;
3. открыть исследование;
4. произвести разведчик;
5. отправить первую миссию;
6. получить первый отчёт;
7. сформировать понятную следующую цель.

Tutorial должен объяснять не расположение кнопок, а причинно-следственные связи.

## 10. Код и архитектура

## 10.1. `main.ts`

Сейчас `main.ts` является допустимым composition root, но начинает перегружаться mount-логикой.

Следующий шаг без сложного framework:

- `ApplicationController`;
- `ScreenRegistry`;
- `CommandBus` или единый typed bridge;
- централизованный lifecycle `mount/dispose`;
- единый state subscription mechanism.

## 10.2. Reducer

`executeCommand` уже содержит большой switch по всем доменам.

Это пока работает, но дипломатия и endgame сильно увеличат файл.

Рекомендуемая структура:

```text
simulation/commands/
  buildingHandlers.ts
  economyHandlers.ts
  researchHandlers.ts
  fleetHandlers.ts
  pveHandlers.ts
  commandHandlers.ts
  diplomacyHandlers.ts
  endgameHandlers.ts
```

Верхний reducer должен выполнять только routing.

Не нужно вводить тяжёлый event-sourcing framework. Достаточно явного handler registry.

## 10.3. Module-level UI state

Некоторые UI-модули хранят состояние в module-level переменных.

Риски:

- повторный mount;
- утечки listener;
- HMR;
- сложные unit tests;
- несколько экземпляров экрана;
- скрытая зависимость между функциями.

Рекомендуется scoped controller:

```ts
class PlanetScreenController {
  private state: GameState;
  private activePlanetId: string;
  private selectedBuildingId?: string;

  mount(): void;
  render(): void;
  dispose(): void;
}
```

## 10.4. Catalog access

Часть кода всё ещё напрямую использует `AEGIS_RESEARCH_CATALOG` и Aegis building lookups.

После завершения #78–#79 необходимо провести отдельный audit:

- нет ли прямых Aegis imports в shared simulation;
- все ли определения идут через faction registry;
- корректно ли создаются стартовые здания каждой фракции;
- сохранены ли старые save IDs;
- не зависят ли UI view models от Aegis assets.

## 10.5. Error messages

Reducer часто возвращает англоязычный `message`, а UI показывает в основном `code`.

Нужно разделить:

- стабильный machine code;
- локализуемый message key;
- structured details;
- player-facing explanation;
- developer diagnostics.

Иначе localization readiness будет дорогой.

## 14. Документация и процесс разработки

## 14.1. Сильные стороны

- source-of-truth порядок зафиксирован;
- PR имеют focus;
- change notes сохраняются;
- есть continuation guide;
- AI может продолжать проект без приватной памяти диалога;
- опасные действия ограничены.

## 14.2. Документация отстаёт от GitHub

На baseline после PR #77:

- `docs/project-status.json` всё ещё может считать #77 активным;
- README описывает более ранний baseline;
- roadmap baseline также отстаёт от фактического `main`.

Это не косметическая проблема. Следующий AI-agent может:

- повторно запланировать уже выполненную работу;
- начать branch от неверного ожидания;
- пропустить migration;
- считать старую проблему актуальной;
- нарушить порядок dependent PR.

### Решение

Добавить CI validation script:

- `lastMergedPr` должен соответствовать последнему merge commit pattern;
- `activePr` не должен быть уже merged;
- README current status должен ссылаться на актуальный milestone;
- roadmap baseline не должен быть старше project-status без явной отметки historical.

## 14.3. Пачки по 6–8 PR оптимизируют throughput, а не качество игры

Большие последовательные batches удобны для автономной поставки, но создают риск механического наращивания scope.

Рекомендуемый новый gate после каждых 2–3 gameplay PR:

1. ручная партия;
2. headless simulation;
3. UX review;
4. performance snapshot;
5. bug triage;
6. только затем следующий mechanics batch.

Обычные code defects не должны считаться внешними blockers, но играбельность должна быть формальным acceptance gate.

## 15. Ассеты, лицензии и публичный релиз

Перед широким публичным релизом следует провести отдельную проверку уникальности названия Stellar Empires:

- search visibility;
- существующие игры;
- домены;
- store naming;
- trademark conflicts.

Даже без юридического конфликта неуникальное название ухудшает поиск и узнаваемость.

## 16. Что не нужно делать

## 16.1. Не переписывать проект на React

Текущие проблемы UI решаются controller lifecycle и registry. Framework migration даст большой объём работы без прямого прироста играбельности.

## 16.2. Не переносить весь UI в Phaser

Canvas неудобен для сложных форм, списков, доступности и текстовых отчётов. Текущий гибрид правильный.

## 16.3. Не добавлять multiplayer

Multiplayer потребует:

- серверного authority;
- accounts;
- sync;
- security;
- anti-cheat;
- moderation;
- deployment infrastructure.

Это отдельный продукт.

## 16.4. Не добавлять новые десятки механик до stabilization

Сначала необходимо доказать:

- корректное время;
- честный AI;
- полную партию;
- устойчивые saves;
- понятный onboarding;
- приемлемый баланс.

## 16.5. Не считать зелёный CI доказательством качества игры

Unit tests подтверждают контракты, но не отвечают:

- интересно ли играть;
- понятен ли интерфейс;
- работает ли pacing;
- не существует ли одной доминирующей стратегии;
- честно ли ощущаются боты.

## 19. Критерии готовности к версии 1.0

Версия 1.0 может считаться готовой только если выполнены все условия.

### Игровой цикл

- есть новая партия с настройками;
- есть понятная ранняя игра;
- есть midgame;
- есть endgame;
- есть победа и поражение;
- есть итоговая статистика.

### Боты

- не читают скрытое состояние;
- корректно догоняют время;
- сохраняют timing и intents;
- используют все обязательные игровые системы;
- проходят полные партии без runtime вмешательства.

### Баланс

- массовые headless runs;
- нет фракции с необъяснимым доминированием;
- нет обязательного единственного build order;
- поражение допускает восстановление;
- партия имеет целевой диапазон длительности.

### Persistence

- schema migrations проверены fixtures;
- save/load не меняет результат;
- save size ограничен;
- recovery протестирован;
- import/export работает между supported browsers.

### UX

- новичок завершает первый цикл без внешней инструкции;
- цели видимы;
- blockers объясняются;
- основные действия не скрыты в случайных dialogs;
- все экраны имеют empty/loading/error states.

### Accessibility

- полная keyboard path;
- DOM-альтернатива canvas;
- focus management;
- reduced motion;
- high contrast;
- автоматический accessibility scan.

### Производительность

- установлен startup budget;
- установлен bundle budget;
- установлен memory budget;
- длинная партия не деградирует неприемлемо;
- low-end fallback проверен.

### Release

- лицензия кода;
- provenance assets;
- credits;
- GitHub Pages QA;
- release tag;
- актуальные README, roadmap и status.

## 20. Финальный вывод

У проекта правильная инженерная база и уже большой объём работающих систем. Переписывание архитектуры или смена движка не требуется.

Главная опасность — продолжить добавлять механику за механикой, не исправив временную модель ботов, не ограничив рост state и не проверив полную партию.

Лучшее решение на текущем этапе:

1. завершить механические каталоги трёх фракций;
2. провести stabilization milestone;
3. встроить AI в каноническое время;
4. добавить ранний complete game loop;
5. начать headless balance намного раньше roadmap #92;
6. после подтверждения играбельности переходить к дипломатии, коалициям и финальному endgame.

**Проекту сейчас нужнее не больше функций, а доказательство, что уже созданные функции образуют честную, понятную и интересную игру.**
