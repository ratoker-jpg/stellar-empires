# Technical Architecture

**Статус:** Draft  
**Последнее обновление:** 2026-07-18

## 1. Цель архитектуры

Архитектура должна поддерживать большую автономную симуляцию в браузере без серверной части и без зависимости результата от частоты кадров.

Главное требование: игровой мир должен существовать как отдельная модель, а Phaser должен только отображать её и передавать команды игрока.

## 2. Зафиксированный стек

- **Phaser 4** — карта, визуальные сцены, объекты и эффекты.
- **TypeScript** — основной язык проекта.
- **Vite** — локальная разработка и production-сборка.
- **HTML/CSS** — сложные панели, таблицы, формы и отчёты.
- **Web Worker** — тяжёлые расчёты симуляции и ботов.
- **IndexedDB** — локальные сохранения.
- **Vitest** — модульные и интеграционные тесты.
- **GitHub Actions** — проверки и деплой.
- **GitHub Pages** — публикация статической сборки.

## 3. Главные слои

```text
┌─────────────────────────────────────┐
│ UI Layer                            │
│ Phaser scenes + HTML/CSS panels     │
└──────────────────┬──────────────────┘
                   │ commands / views
┌──────────────────▼──────────────────┐
│ Application Layer                   │
│ use cases, validation, orchestration│
└──────────────────┬──────────────────┘
                   │ domain commands
┌──────────────────▼──────────────────┐
│ Simulation Domain                   │
│ economy, fleets, combat, research   │
└──────────────────┬──────────────────┘
                   │ state / events
┌──────────────────▼──────────────────┐
│ Persistence                         │
│ IndexedDB, snapshots, migrations    │
└─────────────────────────────────────┘
```

Боты используют Application Layer и Simulation Domain так же, как игрок. Они не меняют состояние напрямую.

## 4. Ограничения зависимостей

Разрешённое направление:

```text
UI → Application → Domain
Persistence → Domain types
Bots → Application commands
```

Запрещено:

- импортировать Phaser в `simulation`;
- обращаться к DOM из доменной логики;
- менять сохранение напрямую из сцены;
- считать экономику в `update()`;
- хранить игровое состояние внутри Phaser Game Objects;
- использовать визуальные таймеры как источник истины.

## 6. Командная модель

Любое действие оформляется командой.

Пример:

```ts
interface UpgradeBuildingCommand {
  type: "UPGRADE_BUILDING";
  empireId: string;
  planetId: string;
  buildingId: string;
  requestedAt: number;
}
```

Обработчик команды:

1. читает текущее состояние;
2. проверяет права и требования;
3. рассчитывает стоимость;
4. изменяет состояние;
5. создаёт доменные события;
6. возвращает структурированный результат.

## 7. Событийная модель

Примеры событий:

- `BUILDING_UPGRADE_STARTED`;
- `BUILDING_UPGRADE_COMPLETED`;
- `RESEARCH_STARTED`;
- `FLEET_DISPATCHED`;
- `FLEET_ARRIVED`;
- `COMBAT_RESOLVED`;
- `COLONY_CREATED`.

Событие должно содержать достаточно данных для журнала, отладки и воспроизводимости.

## 8. Phaser

Phaser отвечает за:

- визуализацию галактики;
- выбор объектов;
- камеры и масштабирование;
- анимации;
- эффекты;
- интерактивные игровые сцены.

Phaser не отвечает за:

- баланс;
- экономические расчёты;
- принятие решений ботами;
- источник игрового времени;
- сохранение;
- финальный расчёт боя.

## 9. HTML-интерфейс

HTML/CSS используется для:

- карточек зданий;
- дерева исследований;
- очередей;
- таблиц кораблей;
- отчётов;
- настроек партии;
- импорта и экспорта сохранений.

Это снижает сложность адаптивной вёрстки и не заставляет имитировать обычный интерфейс средствами canvas.

## 10. Web Worker

В Worker выносятся:

- массовая обработка событий;
- офлайн-пересчёт;
- планирование ботов;
- тяжёлые боевые симуляции;
- прогон ускоренного времени.

Основной поток передаёт Worker снимок или пакет изменений и получает результат. Нельзя передавать в Worker Phaser-объекты или функции.

## 11. Сохранения

Источник истины — сериализуемое состояние мира.

Минимальная структура:

```ts
interface GameSave {
  schemaVersion: number;
  gameVersion: string;
  worldSeed: string;
  createdAt: number;
  savedAt: number;
  simulatedUntil: number;
  settings: GameSettings;
  state: WorldState;
  pendingEvents: ScheduledEvent[];
}
```

Требования:

- автоматическое сохранение;
- несколько слотов;
- атомарная запись;
- резервный предыдущий snapshot;
- миграции между версиями схемы;
- JSON-экспорт и импорт;
- проверка целостности.

## 12. Деплой

Поток:

```text
push / pull request
        ↓
npm ci
        ↓
typecheck + test + build
        ↓
merge в main
        ↓
GitHub Pages deploy
```

Vite должен учитывать базовый путь репозитория `/stellar-empires/`.

## 13. Безопасность и ограничения браузера

- сохранения принадлежат конкретному браузеру и профилю;
- очистка данных сайта удалит IndexedDB;
- перенос между устройствами выполняется JSON-файлом;
- GitHub Pages не предоставляет серверное состояние;
- публичная страница не должна содержать секреты;
- внутриигровые данные и формулы считаются доступными клиенту.

## 14. Производительность

Целевые правила:

- не пересчитывать весь мир каждый кадр;
- использовать очередь событий;
- обновлять UI по изменившимся данным;
- группировать решения ботов;
- ограничивать историю отчётов;
- профилировать массовый офлайн-пересчёт;
- избегать создания большого количества временных объектов в горячих циклах.

## 15. Критерий правильности архитектуры

Доменную симуляцию можно запустить в тестах без браузера, Phaser, canvas и DOM.
